# Финальная ultra-версия ТЗ для ИИ

## Задача: добавить режим списка моделей в каталог авто

---

## 0. Главный принцип работы

Перед любыми изменениями **сначала изучить текущую реализацию**, а уже потом вносить код.

Нельзя придумывать названия полей, структуру фильтров или SQL-логику “с головы”.

Для правильного обозначения полей и фильтров нужно сначала:

1. посмотреть структуру БД;
2. изучить текущий endpoint `GET /catalog`;
3. изучить обработчик `getFilteredCars`;
4. посмотреть, какие поля уже реально используются в существующих запросах к таблице;
5. использовать ту же логику и те же поля в новом endpoint списка моделей.

### Важно

В новом endpoint `/catalog/models` нужно использовать **те же поля и ту же логику фильтрации, что уже используются в `/catalog`**, за исключением одного уточнения:

* для режима моделей брать диапазон лет из поля **`year`**;
* **не** использовать для этого `yearmonth`.

Во всём остальном нужно максимально опираться на уже существующую реализацию `/catalog`.

---

## 1. Цель задачи

На сайте уже есть каталог объявлений.

Нужно добавить **второй режим каталога — список моделей**.

После реализации на странице каталога должно быть 2 режима:

* **объявления**
* **модели**

При этом:

* режим объявлений должен продолжить работать как сейчас;
* режим моделей должен быть новым отдельным сценарием;
* логика моделей **всегда строится через группировку по модели**;
* сортировка моделей **всегда идёт по количеству объявлений по убыванию**;
* это правило одинаково действует **с фильтрами и без фильтров**;
* это правило должно соблюдаться **и на backend, и на frontend**.

---

## 2. Текущая backend-архитектура

Сейчас backend построен на Express и содержит такие routes:

```js
app.get('/filters', getFilters);
app.get('/herocard', heroCard);
app.get('/catalog', getFilteredCars);
app.get('/car/:id', getCarDetails);
app.get('/deliveryCost', deliveryCost);
app.get('/hp', horsePower);

app.post('/lead', submitLead);
app.post('/hp', horsePower);
```

Главный текущий endpoint каталога:

* `GET /catalog` → `getFilteredCars`

Именно он уже знает:

* какие query params приходят с фронта;
* какие поля используются в БД;
* как строится фильтрация;
* как собираются SQL-запросы.

Поэтому новая логика списка моделей должна быть реализована **на основе существующего `/catalog`**, а не как отдельная фантазия с новыми полями.

## 3. Что сделать на backend

### 3.1. Добавить новый endpoint

В файл:

`/root/tarasov-auto/backend/index.js`

нужно добавить новый route:

```js
app.get('/catalog/models', getCatalogModels);
```

### 3.2. Добавить новый handler

Создать новый backend-компонент:

`/root/tarasov-auto/backend/components/getCatalogModels.js`

Импортировать его в `index.js`.

### 3.3. Важное архитектурное правило

Не ломать и не переписывать текущий `GET /catalog`.

Новый режим должен быть реализован через **отдельный endpoint**:

* `/catalog` — список объявлений
* `/catalog/models` — список моделей

## 4. Что сделать на frontend

На месте текущего блока типа:

* `Найдено X объявлений`

нужно сделать **две кнопки-переключателя**:

* `X объявлений`
* `X моделей`

Где:

* `X объявлений` — общее количество объявлений по текущей выборке;
* `X моделей` — общее количество моделей по текущей выборке.

### Поведение

* активной может быть только одна кнопка;
* по умолчанию активен режим **объявлений**;
* при клике на `X объявлений` используется текущая логика каталога;
* при клике на `X моделей` включается новый режим списка моделей;
* визуально кнопки должны соответствовать макету пользователя.

---

## 5. Главная логика режима моделей

### Абсолютное правило

Для режима моделей логика **всегда** такая:

1. сначала применяются фильтры;
2. потом выполняется **группировка по модели**;
3. потом выполняется **сортировка по количеству объявлений по убыванию**;
4. только потом применяется пагинация.

### Это правило действует всегда

* без фильтров;
* с фильтрами;
* на backend;
* на frontend.

### Нельзя делать так

* сначала пагинировать объявления, потом собирать модели;
* сначала брать часть записей, потом агрегировать;
* сортировать модели по цене, бренду, году и т.д. по умолчанию.

---

## 6. Принцип группировки

Группировка списка моделей **всегда** должна выполняться по сочетанию:

* `brand`
* `model`

Нельзя группировать только по `model`, иначе одинаковые названия моделей у разных брендов могут смешаться.

---

## 7. Принцип сортировки

Список моделей **всегда** должен быть отсортирован по количеству объявлений.

То есть:

```sql
ORDER BY ads_count DESC
```

Это обязательное правило:

* в SQL / backend-логике;
* в ответе endpoint;
* в любом frontend-рендере.

Даже если на фронте будет дополнительная локальная обработка списка, итоговый порядок должен оставаться таким же: **по `ads_count DESC`**.

## 10. Логика backend endpoint `/catalog/models`

Новый handler `getCatalogModels` должен работать так:

### Шаг 1. Прочитать query params

Получить:

* `page`
* `pageSize`
* все фильтры каталога

### Шаг 2. Нормализовать параметры

* page >= 1
* pageSize > 0
* выставить значения по умолчанию, если параметры не переданы
* привести числа к числам
* удалить пустые значения

### Шаг 3. Проверить запрещённые фильтры

Если в запросе присутствуют:

* `generation`
* `type`

то запрос списка моделей не должен выполняться как обычный.

Основная защита должна быть на фронте, но backend тоже должен уметь безопасно обработать такой сценарий.

Допустимые варианты реакции backend:

#### Вариант A — вернуть 400

```json
{
  "mode": "models",
  "error": "generation/type filters are not supported in models mode"
}
```

#### Вариант B — вернуть специальный флаг

```json
{
  "mode": "models",
  "modelsModeBlocked": true,
  "reason": "generation/type filters are not supported in models mode"
}
```

### Шаг 4. Построить отфильтрованную базовую выборку

Сначала нужно отфильтровать объявления **по той же логике, что и в `/catalog`**.

Нельзя делать отдельную несовместимую фильтрацию.

### Шаг 5. Выполнить группировку по модели

Группировка:

* `brand`
* `model`

### Шаг 6. Посчитать агрегаты по модели

Для каждой модели нужно вернуть агрегированные данные.

### Шаг 7. Отсортировать

Всегда:

```sql
ORDER BY ads_count DESC
```

### Шаг 8. Применить пагинацию

Пагинация должна применяться **после** фильтрации, группировки и сортировки.

### Шаг 9. Вернуть JSON-ответ

## 11. Требование к переиспользованию логики фильтрации

Нельзя делать две разные логики фильтрации:

* одну в `/catalog`
* другую в `/catalog/models`

Нужно:

1. изучить текущую реализацию `getFilteredCars`;
2. переиспользовать её фильтрационную логику;
3. если нужно — вынести общий helper для сборки filters / WHERE-условий.

### Предпочтительная архитектура

Если фильтрация сейчас зашита прямо в `getFilteredCars.js`, лучше вынести общую часть в helper, например:

* `buildCatalogFilters.js`
* `catalogQueryBuilder.js`
* `applyCatalogFilters.js`

И использовать его:

* в `getFilteredCars`
* в `getCatalogModels`

Цель: одно и то же поведение фильтров в обоих endpoints.

## 12. Какие данные вернуть для каждой модели

Для каждой агрегированной модели вернуть:

### 12.1. `brand`

Марка.

### 12.2. `model`

Название модели.

### 12.3. `ads_count`

Количество объявлений по этой модели внутри **уже отфильтрованной выборки**.

Это главное поле для сортировки.

### 12.4. `photos_preview`

Массив из 4 изображений.

Логика:

* взять первые 4 объявления внутри модели;
* для каждого объявления взять главное фото.

#### Для Китая

Если `photos` приходит строкой вида:

```text
url1,url2,url3
```

нужно брать первую ссылку до запятой.

#### Для Кореи

Если `photos` хранится как JSON / массив объектов, где есть элементы вида:

```json
{
  "code": "001",
  "path": "/carpicture09/pic4139/41391989_001.jpg"
}
```

нужно брать фото, где:

* `code = "001"`

Если в проекте уже есть функция, которая собирает полный URL корейской картинки, использовать именно её.

### 12.5. `year_min` / `year_max`

Считать из поля:

* `year`

Важно:

* **не** использовать `yearmonth` для `/catalog/models`.

### 12.6. `price_min` / `price_max`

Считать из:

* поля, которое уже используется в `/catalog` для цены;
* по описанию это `totalprice_rub`, но финально нужно подтвердить по существующему endpoint.

### 12.7. `displacement_min` / `displacement_max`

Считать из `displacement`.

Преобразование:

* `1900` → `1.9`

Правило округления:

* 1900–1949 → 1.9
* 1950–1999 → 2.0

То есть после перевода в литры округлять до 1 знака по обычной логике.

### 12.8. `fuel_types`

Список уникальных значений `fuel_type` внутри модели.

### 12.9. `hp_min` / `hp_max`

Считать из `hp`.

## 13. Как трактовать фильтры в режиме моделей

Если в `/catalog/models` пришли фильтры, они должны работать так же, как в `/catalog`.

То есть:

* сначала фильтрация объявлений;
* потом группировка моделей по оставшемуся набору.

### Примеры

#### Если пришёл `brand = BMW`

Нужно строить список моделей только внутри BMW.

#### Если пришёл фильтр минимальной цены

Нужно сначала ограничить объявления по этому фильтру, и только потом строить агрегаты моделей.

#### Если после фильтра в модели остались только более дорогие версии

То:

* `price_min/price_max` должны считаться по оставшимся объявлениям;
* `year_min/year_max` тоже по оставшимся;
* `fuel_types`, `hp_min/hp_max`, `photos_preview` тоже по оставшимся.

То есть все агрегаты модели считаются **по отфильтрованной выборке**, а не по всей модели в таблице.

## 14. Пагинация

Endpoint `/catalog/models` должен поддерживать:

* `page`
* `pageSize`

### Важно

Пагинация должна применяться **только после**:

1. фильтрации;
2. группировки по `brand + model`;
3. сортировки по `ads_count DESC`.

Нельзя пагинировать исходные объявления до агрегации.

### В ответе вернуть

* `page`
* `pageSize`
* `totalModels`
* `totalPages`
* `items`

Также желательно вернуть:

* `totalAds`

Чтобы фронтенду было удобно показывать кнопку `X объявлений`.

## 17. Ограничение по `generation` и `type`

Если выбран хотя бы один из фильтров:

* `generation`
* `type`

то режим моделей не должен работать как обычный.

### Причина

Эти поля есть только у корейских машин и неуниверсальны для общего режима моделей.

### Что нужно сделать на frontend

Предпочтительная логика:

#### Сценарий A

Если пользователь в режиме объявлений выбирает `generation` или `type`:

* кнопка `X моделей` становится `disabled`.

#### Сценарий B

Если пользователь уже находится в режиме моделей, а потом выбирает `generation` или `type`:

* интерфейс автоматически возвращает его в режим объявлений;
* запрос на `/catalog/models` не отправляется.

### Что нужно сделать на backend

Даже если фронт не пропустит такой запрос, backend тоже должен безопасно отработать этот случай.

---

## 18. Карточки моделей на frontend

В режиме моделей нужно рендерить карточки **1 в 1 по макету пользователя**.

### Каждая карточка должна содержать

* сетку 2x2 из 4 фото;
* `brand + model`;
* диапазон лет: `year_min - year_max`;
* диапазон цен в рублях;
* строку характеристик:

  * диапазон объёма двигателя;
  * список типов топлива;
  * диапазон л.с.;
* количество объявлений.

### Визуально нужно повторить

* размеры карточек;
* скругления;
* отступы;
* сетку фото;
* порядок информации;
* общий внешний вид максимально близко к макету.

---

## 19. Обновление списка моделей при изменении фильтров

Если активен режим моделей, то при любом изменении поддерживаемых фильтров фронт должен:

1. обновить состояние фильтров;
2. снова вызвать `GET /catalog/models`;
3. получить новый агрегированный список моделей;
4. перерисовать карточки.

### Исключение

Если выбран `generation` или `type`, запрос на `/catalog/models` не отправлять.

## 22. Acceptance criteria

Задача считается выполненной, если:

1. На фронте появились две кнопки:

   * `X объявлений`
   * `X моделей`
2. Режим объявлений работает как раньше.
3. В `index.js` добавлен route `GET /catalog/models`.
4. Создан новый handler `getCatalogModels`.
5. Для моделей логика **всегда**:

   * фильтрация;
   * группировка по `brand + model`;
   * сортировка по `ads_count DESC`;
   * пагинация после агрегации.
6. Это правило работает и без фильтров, и с фильтрами.
7. В `/catalog/models` используются реальные поля и фильтры из существующего `/catalog`.
8. Для диапазона лет в моделях используется `year`, а не `yearmonth`.
9. Карточки моделей визуально соответствуют макету.
10. Если выбран `generation` или `type`, режим моделей не отправляет обычный запрос.
11. И backend, и frontend соблюдают сортировку моделей по количеству объявлений.

---

## 23. Важные запреты

ИИ не должен:

* ломать текущий `GET /catalog`;
* объединять список объявлений и список моделей в один endpoint;
* придумывать новые названия полей без проверки БД;
* придумывать новые query params без проверки `/catalog`;
* использовать другую логику фильтрации, чем в `/catalog`;
* использовать `yearmonth` вместо `year` для диапазона лет моделей;
* группировать модели только по `model` без `brand`;
* сортировать модели по чему-то кроме `ads_count DESC` по умолчанию;
* применять пагинацию до группировки;
* отправлять режим моделей при `generation/type`.

---

## 24. Краткая формула логики

Для `/catalog/models` логика всегда должна читаться так:

```text
берём те же фильтры, что у /catalog
→ фильтруем объявления
→ группируем по brand + model
→ считаем агрегаты модели
→ сортируем по ads_count DESC
→ пагинируем модели
→ возвращаем карточки моделей
```

Это и есть основа всей реализации.
